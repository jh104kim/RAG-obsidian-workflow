주제: $ARGUMENTS
(주제가 비어 있으면 기본 주제 "최첨단 RAG + Obsidian 위키 하이브리드 시스템 구축"으로 진행한다.)

다음 멀티 에이전트 파이프라인을 순서대로 실행하라. 각 단계는 Agent 도구(subagent_type에 해당 에이전트 이름 지정)로 실행하고, 이전 단계의 산출물 전체를 다음 에이전트의 프롬프트에 포함시켜라.

## 파이프라인

### 1단계 — researcher
`researcher` 에이전트에게 주제를 전달해 4개 축(Obsidian 위키 설계 / RAG 파이프라인 / 고급 RAG 기법 / Obsidian-RAG 연동 도구)의 출처 기반 조사를 수행시킨다. 최소 10개 소스, 모든 주장에 URL 근거, 실제 도구/라이브러리 이름 필수.

### 2단계 — analyst
`analyst` 에이전트에게 1단계 조사 결과 전체를 전달해 의사결정 표를 만들게 한다: 접근법별 장단점·트레이드오프·도입비용·난이도, 로컬 무료 구성 vs API 유료 구성 비교, 사용자 스택(Python, GitHub Pages, Obsidian) 맞춤 추천.

### 3단계 — writer
`writer` 에이전트에게 1·2단계 산출물 전체를 전달해 아래 [최종 산출물 형식]대로 초안을 작성시킨다.

### 4단계 — critic ↔ writer 루프 (최대 3회)
1. `critic` 에이전트에게 writer의 산출물 전체를 전달해 심사시킨다. critic은 APPROVED 또는 REVISION NEEDED를 판정하고, 지적마다 심각도(치명/보통/사소)를 표기한다.
2. **APPROVED(치명 지적 0개)면 루프를 즉시 종료**하고 5단계로 간다.
3. REVISION NEEDED면 `writer` 에이전트에게 직전 산출물 + critic의 지적 전체를 전달해 재작성시킨다.
4. **매 라운드 종료 시 사용자에게 `변경점:` 한 줄 changelog를 출력한다** (예: `변경점: [라운드 2] 청킹 코드의 미정의 변수 수정, 로컬 구성안에 임베딩 모델 설치 단계 추가`).
5. 루프는 최대 3회. 3회 후에도 REVISION NEEDED면 마지막 버전을 채택하되, 미해결 치명 지적을 사용자에게 명시한다.

### 5단계 — 파일 저장
최종 산출물 2개를 현재 디렉토리에 파일로 저장한다:
- `RAG-Obsidian-구축-가이드.md` (메인 가이드 A)
- `MOC-RAG-Obsidian.md` (MOC 인덱스 노트 B)
저장 후 사용자에게 파일 경로, 최종 판정, 총 라운드 수를 보고한다.

## [최종 산출물 형식] — writer에게 매번 그대로 전달할 것

### A. 메인 구축 가이드 (.md)
- 상단 YAML frontmatter: `title`, `tags`, `created`(오늘 날짜)
- 핵심 개념은 `[[위키링크]]` 표기 (예: [[RAG]], [[MOC]], [[청킹]], [[벡터DB]])
- 단계별(Stage) 체크리스트(`- [ ]`) + 복붙 가능한 코드블록/명령어
- **"로컬 무료 구성안"과 "API 구성안" 둘 다 제시** (네트워크 제약 환경 고려)
- 마지막에 **"내가 다음에 할 일"** 액션 리스트
- 문서 성격: 읽는 보고서가 아니라 따라 하면 구축되는 핸즈온 매뉴얼

### B. MOC 인덱스 노트 (.md)
- 메인 가이드와 하위 개념 노트들을 `[[링크]]`로 묶는 허브 노트
- YAML frontmatter 포함
