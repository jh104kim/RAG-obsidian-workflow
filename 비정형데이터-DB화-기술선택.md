# 비정형 데이터 DB화 — 기술 선택 가이드

> 2026-06 기준 정리. [[RAG-Obsidian-구축-가이드]]의 확장 노트.
> 실행 계획은 [[TODO-비정형데이터-확장-로드맵]] 참조.

## 1. 기술 선택: 핵심 기준 2가지

**① 데이터 성격 ② 질문 유형**으로 결정한다.

| 상황 | 선택 | 이유 |
|---|---|---|
| 수치 집계·필터·정렬 ("작년 매출 합계는?") | **파싱 → 정형 DB + [[Text-to-SQL]]** | 집계는 LLM이 아니라 SQL이 해야 정확 |
| 의미 검색·요약 ("위약금 조항은?") | **파싱 → 청킹 → [[RAG]]** | 정답이 텍스트 안에 있음 |
| 엔티티 간 관계·다단계 추론 ("A사와 거래한 회사들의 공통 협력사는?") | **[[GraphRAG]] (Graph DB)** | 벡터 검색은 2-hop 이상 관계를 못 따라감 |
| 둘 다 섞임 (실무 대부분) | **하이브리드 + 라우터** | LLM이 질문을 분류해 SQL/RAG/Graph로 분기 |

핵심: **4가지는 경쟁 관계가 아니라, 파싱이 공통 전처리이고 나머지는 질문 유형별 분기다.**

## 2. 파일 형식별 파싱 전략

| 형식 | 권장 도구 | 처리 방향 |
|---|---|---|
| 비정형 엑셀 (병합셀, 다중 헤더) | openpyxl + LLM 정규화, 또는 [[Docling]] | 표 구조 복원 → tidy 형태(1행=1레코드)로 펴서 RDB 적재 |
| PDF (디지털) | **Docling** (IBM), MinerU | 표 → DB, 본문 → RAG 청크로 분리 |
| PDF (스캔/이미지) | OCR(PaddleOCR) 또는 VLM 직접 파싱 | 최근엔 OCR보다 VLM(GPT-4o/Claude에 이미지 투입)이 표 복원 정확도 높음 |
| PPT | python-pptx 추출 + 슬라이드 이미지 VLM 캡셔닝 | 슬라이드는 시각 정보가 핵심, 텍스트만 뽑으면 손실 큼 |
| 그림/차트 | [[ColPali]]/ColQwen 계열 | 페이지를 이미지 그대로 임베딩 → 파싱 자체를 생략하는 멀티모달 RAG |

- *Docling: 문서 레이아웃·표를 구조화 JSON/Markdown으로 추출하는 오픈소스 파서*
- *ColPali: 문서 페이지 이미지를 직접 벡터화해 텍스트 추출 없이 검색하는 모델*

## 3. 쿼리 정확도 평가 방법

### Text-to-SQL

| 지표 | 의미 |
|---|---|
| **EX (Execution Accuracy)** | 생성 SQL의 실행 결과가 정답 결과와 일치하는지. 표준 지표 |
| EM (Exact Match) | SQL 문자열 일치. 같은 답도 다른 SQL이 가능해 거의 안 씀 |
| Test-suite accuracy | 여러 DB 인스턴스에서 실행해 우연한 일치 제거 |

**현실 체크 (2026-06, BIRD 벤치마크)**: 테스트 1위 ~82% vs 인간 93%. 멀티턴 상호작용 평가(BIRD-Interact)에선 최고 모델도 ~16%.
→ **Text-to-SQL은 아직 100% 신뢰 불가. 검증 루프 필수.**

### RAG

- [[RAGAS]] 4지표: faithfulness, answer relevancy, context precision/recall
- 검색 단계 분리 측정: Hit@k, MRR, nDCG
  - *nDCG: 상위 랭킹에 정답 문서가 있을수록 높은 점수를 주는 검색 품질 지표*

### 공통

평가셋 없으면: LLM 합성 Q&A 생성(RAGAS testset generator) → LLM-as-judge 채점 → 사람이 샘플 검수.

## 4. 최근 정확도 향상 기법

### Text-to-SQL
- **후보 다중 생성 + 선택**: SQL N개 생성 후 실행 결과로 다수결/선택 (CHASE-SQL, XiYan-SQL). 효과 가장 큼
- **Schema linking**: 질문 관련 테이블·컬럼만 추려 프롬프트 투입
- **실행 기반 self-correction**: 실행 에러/빈 결과 시 에러 메시지 보고 재생성
- **시맨틱 레이어**: 컬럼 설명·용어 사전·대표값 메타데이터 제공. 실무 체감 효과 최대
- **Agentic 멀티턴**: 모호한 질문은 되묻기. 2026년 벤치마크 트렌드가 단발 생성 → 상호작용으로 이동 중

### RAG
- 하이브리드 검색(BM25+벡터) + 리랭커 — [[RAG-Obsidian-구축-가이드]]에 반영됨
- **Query rewriting/decomposition**: 질문 재작성 또는 하위 질문 분해
- **Agentic RAG**: 검색 결과 부족 시 스스로 재검색 판단
- **GraphRAG**: 엔티티·관계 추출 → 그래프 + 커뮤니티 요약으로 전체 조망형 질문 대응. 구축 비용(LLM 호출) 커서 관계형 질문이 실제로 많을 때만

## 5. 추천 아키텍처

```
파일 입수 → Docling 파싱 → ① 표 → SQLite/DuckDB
                          ② 본문 → Chroma (기존 RAG)
                          ③ 이미지·차트 → VLM 캡션 후 ②에 합류
질문 → LLM 라우터 → SQL 경로(집계) / RAG 경로(의미) → 답변 + 출처
```

**검토 의견**: GraphDB는 1단계에서 제외. 관계형 질문 빈도 확인 후 추가 (유지비 큼).
평가는 질문 30~50개 골든셋부터 만들고 EX(SQL)·RAGAS(RAG)를 경로별로 따로 측정.
